Code Spire / System Architecture

The system as layers & dependencies

Reads top to bottom, one swimlane per layer: a review is triggered, handled by a few deployable services that talk only through a Kafka / Redpanda bus, build on a single row of shared libraries (the plugin SPI + adapters), persist to data stores, and reach out to external systems. Solid = built today. Dashed = on the roadmap, not yet built (RAG / vector index, repo-rules context, MinIO, OIDC, real webhooks). The board scrolls sideways.

How to read it. One swimlane per layer, top to bottom, with the data stores on their own dedicated lane and every library on a single row. The three services never call each other directly — they publish and subscribe on the Redpanda / Kafka topics (cs.integration → cs.commands → cs.events → cs.results, with cs.dlq for poison messages). spire-orchestrator owns the append-only event store, the read models, and the REST + WebSocket API the dashboard speaks to; spire-review-worker runs the diff → context → review → comment pipeline and loads the plugin libraries (each an SPI port: DiffSource, ContextProvider, LlmProvider, CommentSink) that call out to the external systems lined up directly beneath them. Every module compiles against spire-contract, and credentials + assembled context are sealed with spire-encryption (Tink) before they touch Postgres or the bus. Today's live entry is the manual Register PR call on the orchestrator — the webhook ingress on spire-gateway is built but parked until there's a public host.

Roadmap (dashed). Whole-repo RAG is the largest planned lever: a spire-indexer maintains a vector index (Qdrant / LanceDB) from repo pushes, and a spire-context-rag provider retrieves snippets into the same context fan-out — added as a new subscriber with zero change to the review flow, exactly like Jira and Confluence were. A spire-context-rules provider (repo .codespire config), MinIO as the object BlobStore (blobs live in Postgres today), and Keycloak / OIDC on the dashboard round out the near-term backlog.
